# Autoryzacja 2.0 (Aktywacja & Reset Hasła)

## 1. Opis Biznesowy
Funkcja "Autoryzacja 2.0" rozszerza podstawowy system logowania i rejestracji o dwa kluczowe mechanizmy:
*   **Miękka Aktywacja Konta (Soft Activation):** Użytkownik po rejestracji nie jest blokowany. Może wejść do panelu (Dashboard) i zapoznać się z systemem. Jednakże, krytyczne akcje (np. zlecenie windykacji) są zablokowane do czasu potwierdzenia adresu e-mail poprzez link aktywacyjny.
*   **Reset Hasła:** Standardowy przepływ "Zapomniałem hasła", pozwalający użytkownikowi na wygenerowanie linku z tokenem, który umożliwia bezpieczne ustawienie nowego hasła.

## 2. Architektura (Adapter Pattern)
Zgodnie z założeniami projektu, logika autoryzacji jest odseparowana od interfejsu użytkownika. Komponenty React (Next.js) komunikują się wyłącznie z interfejsem `IAuthService`. 

Obecnie implementacją tego interfejsu jest `AppsScriptAuthAdapter`, który deleguje zadania (generowanie tokenów, wysyłka e-maili) do Google Apps Script. W przyszłości, przy przejściu na docelowy backend (np. Supabase/Node.js), wystarczy podmienić adapter, a kod UI pozostanie nietknięty.

### Przepływ Danych (Data Flow)

#### Rejestracja i Aktywacja
1. Użytkownik rejestruje się przez `AuthModals.tsx`.
2. Backend (Apps Script) tworzy konto ze statusem `emailVerified: false` i wysyła e-mail.
3. Użytkownik zostaje automatycznie zalogowany, widzi `VerificationBanner`.
4. Kliknięcie w link w mailu kieruje do `/auth/verify`.
5. Strona aktywacji wywołuje API, które zmienia status w bazie.
6. Przy następnym logowaniu (lub odświeżeniu sesji), banner znika.

#### Reset Hasła
1. Użytkownik klika "Zapomniałeś?" w modalu logowania.
2. Podaje e-mail -> wywołanie `requestPasswordReset`.
3. Otrzymuje e-mail z linkiem do `/auth/reset-password?token=...`.
4. Na stronie resetu podaje nowe hasło.
5. Po sukcesie może zalogować się nowymi danymi.

## 3. Mapowanie Plików (Code Map)

### Warstwa Abstrakcji i Adaptery
*   `src/shared/api/interfaces/IAuthService.ts` - Definicja kontraktów: `requestPasswordReset`, `resetPassword`, `activateAccount`, `resendActivationEmail`.
*   `src/shared/api/adapters/AppsScriptAuthAdapter.ts` - Implementacja wywołań do Apps Script.
*   `src/context/AppContext.tsx` - Rozszerzenie modelu `LoggedInUser` o flagę `emailVerified?: boolean`.

### Interfejs Użytkownika (UI)
*   `src/features/auth/ui/AuthModals.tsx` - Modal logowania i formularz "Zapomniałem hasła".
*   `src/app/auth/verify/page.tsx` - Widok weryfikacji tokenu aktywacyjnego.
*   `src/app/auth/reset-password/page.tsx` - Widok ustawiania nowego hasła.
*   `src/views/dashboard/ui/VerificationBanner.tsx` - Pasek ostrzegawczy dla niezweryfikowanych użytkowników.
*   `src/views/dashboard/ui/DashboardView.tsx` - Logika blokowania przycisków akcji.

### Backend (Google Apps Script)
*   `src/integration/skrypty_GAS/auth_v2_2.gs` - Logika weryfikacji tokenów, aktualizacji statusu, przywrócona funkcja `checkEmailStatus` oraz nowa funkcja `changePassword` do zmiany hasła przez zalogowanego użytkownika.
*   `src/integration/skrypty_GAS/email_templates_v2.gs` - Szablony e-mail dla aktywacji i resetu (wersja stabilna).
*   `src/integration/skrypty_GAS/main_v4.gs` - Rejestracja nowych akcji API (w tym `CHANGE_PASSWORD`) oraz przywrócona obsługa Sitemap XML.
*   `src/integration/skrypty_GAS/provisioning_v2_2.gs` - Obsługa rejestracji z generowaniem tokenów aktywacyjnych, przywrócona inicjalizacja silosów (`initClientSpreadsheet`) oraz logika zmiany nazw na Drive.
*   `src/integration/skrypty_GAS/initialization_v4.gs` - Automatyczna konfiguracja struktury Google Drive i arkuszy dla nowych wdrożeń.

## 4. Katalog Funkcji Technicznych

### `resendActivationEmail`
- **Opis:** Wysyła żądanie o ponowne wygenerowanie i wysłanie linku aktywacyjnego.
- **Parametry:** `{ email: string }`

### `activateAccount`
- **Opis:** Przesyła token do backendu w celu oznaczenia konta jako zweryfikowane.
- **Parametry:** `{ token: string }`

### `requestPasswordReset`
- **Opis:** Inicjuje proces resetowania hasła (generowanie tokenu i wysyłka maila).
- **Parametry:** `{ email: string }`

### `resetPassword`
- **Opis:** Finalizuje zmianę hasła na podstawie tokenu.
- **Parametry:** `{ token: string, newPassword: string }`

### `changePassword`
- **Opis:** Umożliwia zalogowanemu użytkownikowi samodzielną zmianę hasła. Wymaga podania aktualnego hasła w celu weryfikacji tożsamości przed dokonaniem zmiany w bazie.
- **Parametry:** `{ email: string, currentPassword: string, newPassword: string }`
- **Zwraca:** `{ status: 'success', message: 'Hasło zostało pomyślnie zmienione.' }` lub `{ status: 'error', message: 'Błędne aktualne hasło.' }`
